# Itera Bot — Сравнение архитектур: n8n → Python

> Документ для команды и наставников.
> Описывает, что изменилось при миграции Telegram-бота Itera с n8n-workflow на Python-приложение, и почему это важно для продукта.

---

## 1. Стек: было → стало

| Компонент | n8n-версия | Python-версия |
|-----------|-----------|---------------|
| **Среда выполнения** | n8n Cloud (SaaS, визуальный конструктор) | Python 3.11 + Docker-контейнер |
| **Telegram-интеграция** | n8n Telegram Trigger (webhook-нода) | aiogram 3.x (async framework) |
| **База данных** | Supabase REST API (HTTP-запросы из нод) | asyncpg — прямое TCP-подключение с connection pool |
| **LLM-интеграция** | n8n OpenAI-нода (синхронно, без retry) | openai SDK (async, 3 retry с exponential backoff) |
| **FSM (состояние диалога)** | Поле в Supabase + логика в JSON-нодах | Кастомный FSM Storage на asyncpg + aiogram States |
| **Планировщик (напоминания)** | Отдельный n8n-workflow по cron | APScheduler (встроен в приложение) |
| **Деплой** | n8n Cloud (привязка к платформе) | Dockerfile — запускается на любом сервере |
| **Код** | JSON-workflow (нечитаемый, ~2000 строк JSON) | Python (~1500 строк читаемого кода) |
| **Версионирование** | Экспорт JSON-файлов вручную | Git — полноценная история изменений |

## 2. Сравнение по ключевым параметрам

### 2.1 Стабильность работы

| Критерий | n8n | Python | Разница |
|----------|-----|--------|---------|
| Обработка ошибок LLM | Workflow падает целиком. Пользователь не получает ответа | 3 retry, при финальной ошибке — сообщение «попробуй позже», стейт сбрасывается | Пользователь никогда не «зависает» |
| Дубли Telegram-апдейтов | Нет защиты — дубль = дубль записи в БД | Таблица `processed_updates`, идемпотентная вставка | Нет дублей данных |
| Перезапуск сервиса | FSM-состояние теряется, пользователь застревает | FSM хранится в PostgreSQL, переживает рестарт | Бесшовный перезапуск |
| Timeout OpenAI | Workflow висит до таймаута n8n (5 мин), пользователь ждёт | 60 сек таймаут + retry + graceful fallback | Предсказуемое поведение |
| Мониторинг | Логи n8n Cloud (ограниченные) | Полные Python-логи, можно подключить Sentry/Grafana | Полная наблюдаемость |

### 2.2 Масштабируемость

| Критерий | n8n | Python | Разница |
|----------|-----|--------|---------|
| Параллельные пользователи | Последовательная обработка. 5 человек = очередь | Полностью async. 50+ одновременных запросов | 10x+ пропускная способность |
| Подключение к БД | Каждый шаг = новый HTTP-запрос к Supabase REST (100-300ms) | Connection pool (2-10 соединений), ~5-10ms на запрос | 20-50x быстрее на БД |
| Горизонтальное масштабирование | Невозможно (один инстанс n8n) | Несколько контейнеров за балансировщиком | Линейный рост |
| Rate limits | n8n Cloud имеет свои лимиты на выполнения | Ограничен только OpenAI API и ресурсами сервера | Контролируемо |

### 2.3 Потенциал для изменений и допиливания

| Что нужно сделать | n8n: как это выглядит | Python: как это выглядит |
|-------------------|-----------------------|--------------------------|
| Добавить новую кнопку в меню | Найти нужную ноду в визуальном графе, вписать в JSON, протестировать вручную | Добавить строку в `keyboards/main_menu.py`, написать хендлер | 
| Изменить логику check-in | Перетаскивать ноды, менять связи, рисковать сломать соседние ветки | Изменить одну функцию в `handlers/checkin.py` |
| Добавить новый AI-режим | Дублировать целую ветку workflow, переподключать ноды | Создать файл `services/new_mode_ai.py` + добавить `if` в хендлере |
| Написать тесты | Невозможно | pytest + моки — стандартная практика |
| Code review | Сравнивать два JSON по 2000 строк | `git diff` — видны конкретные изменения |
| Подключить аналитику | Внешний сервис + webhook-нода + ручная склейка | `import analytics` + вызов в нужном месте |
| Добавить веб-дашборд | Отдельный проект с нуля | FastAPI-эндпоинты рядом с ботом, общая БД |
| A/B тест промптов | Ручное переключение между workflow | Конфиг/флаг в коде, разные промпты по условию |

### 2.4 Потенциал для пивота

| Сценарий | n8n | Python |
|----------|-----|--------|
| **Сменить Telegram на веб-приложение** | Переписывать с нуля — вся логика зашита в Telegram-ноды | Сервисный слой (`services/`) не зависит от Telegram. Подключить FastAPI = дни, не недели |
| **Сменить OpenAI на Claude/Llama** | Менять каждую AI-ноду вручную | Заменить одну функцию в `llm_client.py` |
| **Добавить второй язык (EN)** | Дублировать все workflow | Добавить параметр `language` в промпты и форматтеры |
| **Монетизация (подписки)** | Очень сложно — нет нормальной авторизации | Добавить поле `subscription` в Profiles + middleware проверки |
| **Командный режим (несколько пользователей на цель)** | Архитектурно невозможно в n8n | Новая таблица + связи — расширение существующей модели |
| **Мобильное приложение** | С нуля, без переиспользования логики | Backend уже есть — добавить REST API поверх тех же сервисов |

## 4. Структура проекта (для разработчиков)

```
itera/
├── bot/
│   ├── main.py              # Точка входа: webhook (prod) / polling (dev)
│   ├── config.py            # Настройки из .env через pydantic
│   ├── database.py          # Все SQL-запросы в одном месте
│   ├── fsm/
│   │   ├── states.py        # Состояния диалога (check-in, цель, профиль...)
│   │   └── db_storage.py    # Хранение состояний в PostgreSQL
│   ├── handlers/            # Обработчики команд и кнопок (8 файлов)
│   ├── keyboards/           # Inline-клавиатуры Telegram
│   ├── services/            # AI-логика и планировщик
│   └── utils/               # Форматирование и идемпотентность
├── migrations/              # SQL-миграции
├── Dockerfile               # Контейнеризация
├── docker-compose.yml       # Оркестрация
├── requirements.txt         # Зависимости Python
└── .env                     # Секреты (не в Git!)
```

## 5. Цифры

| Метрика | n8n | Python |
|---------|-----|--------|
| Строк кода / конфига | ~2000 строк JSON (нечитаемый) | ~1500 строк Python (читаемый) |
| Время ответа на /start | 2-4 сек | 0.5-1 сек |
| Время check-in (с LLM) | 10-20 сек | 8-15 сек |
| Параллельных пользователей | 1 (очередь) | 50+ |
| Время добавления фичи | Часы-дни (визуальный граф) | Минуты-часы (код) |
| Стоимость хостинга | n8n Cloud ~$20+/мес | VPS ~150 руб/мес (~$1.5) |
| Vendor lock-in | Полный (n8n) | Нет (стандартный Python) |

---

## 6. Вывод

Миграция с n8n на Python — это переход от **прототипа** к **продукту**:

- **Для пользователя** — бот работает так же, но стабильнее и быстрее
- **Для разработчика** — код читаемый, тестируемый, расширяемый
- **Для бизнеса** — дешевле в хостинге, нет vendor lock-in, можно масштабировать и пивотить
- **Для команды** — Git, code review, CI/CD — стандартные инструменты разработки

n8n отлично подходил для валидации идеи. Python-версия — для того, чтобы эту идею масштабировать.
